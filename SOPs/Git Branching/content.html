Git Branch Topology

Empat tipe cabang utama yang digunakan di Cube Studio. Patuhi arsitektur ini untuk mencegah conflict fatal.

Production

main / master

Cabang paling sakral. Berisi kode yang 100% stabil dan siap rilis ke pemain/klien.

  • ■ Protected: Dilarang keras melakukan direct commit/push.
  • ■ Sumber: Hanya menerima PR dari staging atau hotfix.
  • ■ Trigger: Memicu Auto Prod Build jika di-tag Release.
Integration

staging / develop

Pusat integrasi kode harian. Tempat berkumpulnya seluruh fitur dari berbagai programmer.

  • ■ Default: Area utama QA Tester melakukan uji coba.
  • ■ Aturan: Dilarang coding berhari-hari langsung di cabang ini.
  • ■ Trigger: Target utama untuk tombol Manual Staging Build.
Working

feature/*

Area kerja individu. Format: feature/[nama-tugas].

  • ■ Asal: Selalu dibuat bercabang dari staging.
  • ■ Tujuan: Setelah selesai, buat PR untuk di-merge ke staging.
  • ■ Sinkronisasi: Wajib rutin pull dari staging jika pengerjaan lama agar tidak conflict.
Emergency

hotfix/*

Cabang darurat untuk memperbaiki bug kritis di versi rilis.

  • ■ Asal: WAJIB dibuat bercabang langsung dari main.
  • ■ Tujuan: Di-merge ke main DAN ke staging.
  • ■ Alasan: Agar bug yang sama tidak terulang pada perilisan fitur di masa depan.

Siklus Pengembangan Fitur

Langkah demi langkah cara mengerjakan tugas baru hingga masuk ke branch integrasi.

1. Persiapan
Checkout & Pull 'staging' terbaru
2. Isolasi Pekerjaan
Buat branch 'feature/[nama-fitur]'
3. Pengerjaan
Coding, Commit, dan Push ke remote
4. Code Review
Buka Pull Request (PR) ke 'staging'
5. Integrasi Selesai
Merge ke 'staging' & Hapus branch fitur

Siklus Darurat (Hotfix)

Alur spesifik saat terjadi bug kritis di versi rilis publik (Production).

1. Titik Awal
Checkout dari 'main'
2. Isolasi Darurat
Buat branch 'hotfix/[nama-bug]'
3. Eksekusi Fix
Perbaiki bug, Commit, Push
4. Code Review
Buka PR ke 'main' & 'staging'
Prioritas 1
Merge ke 'main' (Rilis patch)
Prioritas 2
Merge ke 'staging' (Cegah regressi)
