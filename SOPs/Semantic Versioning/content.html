Anatomi Semantic Versioning (SemVer)

Semua rilis dan tag di GitHub WAJIB mengikuti format standar: v[MAJOR].[MINOR].[PATCH]-[STAGE]

[MAJOR]

Perubahan Arsitektur Besar

Perubahan arsitektur besar-besaran, perombakan core gameplay, atau pembaruan engine yang membuat versi ini tidak kompatibel dengan versi sebelumnya.

[MINOR]

Penambahan Fitur Baru

Penambahan fitur baru yang berjalan lancar dengan sistem yang sudah ada (backward-compatible).
Contoh: Menambahkan sistem inventory baru atau integrasi multiplayer.

[PATCH]

Perbaikan Bug (Bug Fixes)

Perbaikan bug dan hotfix darurat yang tidak mengubah fitur utama.
Contoh: Memperbaiki UI yang glitch, menutup memory leak, dsb.

💡

Contoh Penggunaan

Versi v1.2.4 berarti: Major versi 1, penambahan fitur Minor ke-2, dan Patch/perbaikan bug ke-4.

Tahapan Rilis (Release Stages)

Suffix/akhiran yang menandakan tingkat kematangan dan stabilitas sebuah build. Sangat penting untuk konfigurasi rilis di GitHub.

-alpha

Alpha Stage

Definisi: "Fitur Belum Lengkap". Core mechanic bisa dimainkan, namun sistem utama lainnya mungkin kosong dan sangat rentan bug.
Target: Internal testing tim & prototyping.
Contoh Tag: v0.1.0-alpha, v1.0.0-alpha
Jika menggunakan GitHub: Centang "Set as a pre-release"

-beta

Beta Stage

Definisi: "Fitur Lengkap". Fitur berhenti bertambah. 100% fokus pada perbaikan bug, optimasi, dan pemolesan (polish).
Target: QA Tester, eksternal tester, client preview.
Contoh Tag: v1.0.0-beta, v1.2.0-beta
Jika menggunakan GitHub: Centang "Set as a pre-release"

-rc

Release Candidate

Definisi: "Kandidat Rilis Final". Build dinilai stabil. Jika QA tidak menemukan Game Breaking Bug, build yang sama ini akan jadi Official Release.
Target: Final QA Pass.
Contoh Tag: v1.0.0-rc
Jika menggunakan GitHub: Centang "Set as a pre-release"

Tanpa Suffix

Stable / Official Release

Definisi: "Production Ready". Versi stabil, sudah melewati seluruh tahap QA, dioptimasi penuh, dan siap diluncurkan secara resmi.
Target: Publik / Klien (End-user).
Contoh Tag: v1.0.0, v2.1.3
Jika menggunakan GitHub: Centang "Set as the latest release"

✓

Eksekusi Official Release (dari branch main atau master)

  1. Tentukan Nomor Versi Analisis apa saja yang baru di-merge dari staging. Apakah ini sekadar perbaikan bug (naikkan Patch), ada fitur baru (naikkan Minor), atau perombakan besar (naikkan Major)?
  2. Draft Release Buka repository GitHub, klik tab Releases, lalu pilih Draft a new release.
  3. Buat Tag Ketik nama tag sesuai format SemVer pada kolom "Choose a tag" (Contoh: v1.0.0-beta), lalu klik Create new tag.
  4. Pastikan Target Branch Pastikan opsi target mengarah ke branch main.
  5. Tulis Release Notes Tuliskan dengan jelas penambahan fitur atau bug apa saja yang diperbaiki di versi ini untuk memudahkan dokumentasi.
  6. Atur Status Rilis (Contoh Implementasi GitHub)
    • Jika mem-publish di platform seperti GitHub dan ujung versinya -alpha, -beta, atau -rc, disarankan untuk mencentang kotak "Set as a pre-release".
    • Jika ini versi Stabil (tanpa suffix), centang kotak "Set as the latest release".
  7. Publish Release Klik tombol hijau Publish release. Proses CI/CD Production akan otomatis berjalan di *runner* dan nantinya menempelkan file .zip hasil *build* ke halaman rilis tersebut.
